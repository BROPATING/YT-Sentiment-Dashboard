§ 01

Architectural pillars

Three principles that override every other engineering preference. When in doubt, return to these.

PILLAR 01

Decouple, always

Ingestion, classification, dashboard, and reply-posting are independent services on a message bus. A failure in one does not cascade. A spike in one does not starve the others.

PILLAR 02

Compliance is not a feature

OAuth scopes, data retention, audit logs, and human-in-the-loop approval are baked into the architecture, not added before launch. The YouTube API audit assumes them by default.

PILLAR 03

Cost is a first-class metric

YouTube quota and LLM tokens are budgets, not infinite. Every service accounts for its consumption. The Intelligence Layer is the single choke-point for swapping providers.

§ 04

Data flow: comment lifecycle

From "creator hits publish" to "alert lands in their pocket" — what every comment does in our system.

FIG 02Comment lifecycle · 9 steps from publish to creator notification

The why behind each step

  • Polling, not push. YouTube's PubSubHubbub fires only for new videos, not comments. We poll. The 15-minute interval is a balance between alert latency and quota burn.
  • Dedupe before bus. Cheaper than dedup downstream and avoids fan-out amplification.
  • Two events, not one. comment.ingested and comment.classified are separate so dashboards can render raw comments instantly while classification catches up.
  • Sliding-window toxicity. Single toxic comment ≠ alert. Rolling 1-hour window weighted by comment velocity = alert. Reduces noise dramatically.
§ 05

OAuth & identity

The single highest-stakes flow in the system. Get this wrong and we lose creator trust, YouTube approval, or both.

Scope minimisation

We request only what each capability needs, and we request it lazily — i.e. youtube.force-ssl is only requested when a creator first enables reply-posting, not at signup.

ScopeWhen requestedWhy
openid · email · profileSign-inIdentify the user account.
youtube.readonlyChannel connectList videos, fetch comment threads, fetch channel stats.
youtube.force-sslLazy — first replyPost replies, hide spam, moderate. Only after creator enables Action Center.
yt-analytics.readonlyLazy — Brand ReportAudience health export, sponsor reports.

Token storage

  • Access tokens: never persisted. Held in memory or short-lived Redis (TTL = expiry).
  • Refresh tokens: encrypted with envelope encryption — KMS-managed root key wraps a per-tenant data key, which wraps the token. Stored in oauth_tokens.
  • On revoke or account delete: tokens revoked at Google AND the row hard-deleted within 30 days.
Audit-critical

If a YouTube auditor asks "show me how a deleted user's tokens are guaranteed gone," your answer must be: a runnable script that queries every storage layer (Postgres, Redis, backups, logs) and proves zero residue. Build it on day one. Run it monthly.

§ 06

The intelligence layer

Your competitive moat lives here. Treat it as a product, not a folder.

Why a router, not direct calls

Every model call (sentiment, toxicity, summary, draft, persona) goes through the LLM Gateway — a thin internal service that:

  • Picks the right backend (self-hosted classifier vs. Perspective API vs. Claude vs. GPT) based on cost, latency, and task.
  • Caches identical inputs (a "First!" comment classified once, served forever).
  • Tracks per-tenant token spend and enforces per-plan caps.
  • Logs every call to llm_calls with model version, prompt template ID, cost, latency — feeds the Trust Ledger.
  • Supports a "shadow mode" where we run two models in parallel to compare quality before switching.

Model selection by task

TaskBackendReasoning
Sentiment (bulk)Self-hosted, fine-tuned1000s of comments / minute. API costs would kill margin.
ToxicityPerspective API + internalPerspective is free, well-calibrated for English. Internal model fills Hinglish gap.
Vibe SummaryClaude / GPT (premium)Once per week per creator. Quality matters more than cost. Customer-visible output.
Reply draftsCheap LLM (Haiku-class)High volume, short outputs. Cost-sensitive.
Persona matchingEmbeddings + few-shotRetrieve N nearest approved replies via pgvector, prepend as few-shot.

Evaluation harness (non-optional)

An internal CLI tool, creatorcore-eval, runs the entire labelled comment set against any candidate model and outputs F1 per language, confusion matrix, and cost-per-1k-comments. No model goes to production without passing. This is your scientific moat.

Multi-tenancy strategy

Single shared Postgres database, every tenant-scoped table carries a tenant_id with row-level security. We get isolation guarantees without per-tenant DBs to operate. We revisit at 1000+ tenants or when a single creator exceeds 5% of total comment volume.

API contract

REST + JSON over HTTPS. Versioned at the URL. WebSocket only for live alert push.

Public endpoints (excerpt)

# Channel connection
POST   /v1/channels/connect          # initiates OAuth, returns auth URL
GET    /v1/channels                  # list connected channels
DELETE /v1/channels/{id}             # revoke + purge

# Dashboard reads (cached aggressively)
GET    /v1/channels/{id}/stats?range=7d
GET    /v1/channels/{id}/sentiment-timeseries?range=30d
GET    /v1/channels/{id}/health-score
GET    /v1/videos/{id}/comments?sentiment=negative&page=1

# Reply workflow
GET    /v1/replies/pending           # queue of drafted replies
POST   /v1/replies/{id}/approve      # one-tap publish
POST   /v1/replies/{id}/discard
PATCH  /v1/replies/{id}              # edit before approve

# Insights
GET    /v1/channels/{id}/vibe-summary?week=2026-W17
GET    /v1/channels/{id}/alerts?status=open

# Live updates
WSS    /v1/stream/alerts             # server pushes alert events

Conventions that save future arguments

  • All list endpoints paginate via cursor (not offset). Stable under inserts.
  • All timestamps are RFC 3339 UTC. Always.
  • All errors follow RFC 9457 Problem Details. No bespoke error formats.
  • Idempotency keys on every POST that triggers external side-effects (reply post, billing).
§ 09

Security & compliance

Threat model (top 5)

  1. OAuth token theft → KMS envelope encryption, short-lived access tokens never persisted, monthly key rotation.
  2. Tenant data leakage via SQL bug → Row-level security policies enforced in Postgres + tenant-id assertion in every API handler.
  3. Prompt injection in comments → Comments are data in LLM prompts, never instructions. Strict separation, no eval-style execution. Output filtering on Vibe Summaries.
  4. Stolen JWT → 15-minute access TTL, refresh rotation, anomaly detection on geo + device.
  5. Insider abuse → All admin reads against tenant data are audit-logged with justification; randomised access reviews quarterly.

Mentor's lever

Pass the YouTube API Compliance Audit before you scale. It's slow (4–8 weeks) but once you have it, your story to creators changes from "trust us" to "Google verified us." That's worth more than any feature on your roadmap.

§ 10

Quota & cost engineering

Two budgets that decide whether the company survives: YouTube API quota units, and LLM tokens.

YouTube quota strategy

  • Default daily quota is 10,000 units. commentThreads.list costs 1 unit per call (typically 100 comments returned).
  • Polling math: 1 channel × 96 polls/day × ~3 calls/poll ≈ 288 units/day. 10K quota = ~35 channels max at default.
  • Apply for quota extension at 25 channels, not at exhaustion. Approval takes weeks.
  • Per-tier polling cadence: Free = 60 min, Creator = 15 min, Manager = 5 min. Plan tiers are quota tiers.
  • Aggressive Redis caching: video metadata (1 hour), channel stats (15 min), comments by ID (forever — they don't change).

LLM cost strategy

  • Per-tenant monthly token cap, enforced at the LLM Gateway. Soft warn at 80%, hard cut at 100%.
  • Vibe Summary uses a pre-aggregation step in our own code (cheap) to feed the LLM only top themes — not 5,000 raw comments.
  • Reply drafts use cached templates for the top-N most common low-intent patterns ("First!", "🔥🔥🔥") — zero LLM cost.
  • Per-tier model selection: Free uses templated replies only, paid tiers unlock LLM-drafted personalised replies.

§ 11

Deployment topology

Choices

  • Cloud: Google Cloud Platform, region asia-south1 (Mumbai). Closest to early users + cheap intra-region traffic.
  • Compute: Cloud Run for stateless services (auto-scales to zero — saves money in early days). GKE Autopilot for the classifier (GPU node pool, persistent).
  • Database: Cloud SQL for Postgres (HA, automated backups, point-in-time recovery).
  • Cache: Memorystore for Redis.
  • Bus: Cloud Pub/Sub. Pull-mode subscribers, exactly-once where it matters (reply posts).
  • Edge: Cloudflare in front of everything for WAF, DDoS, CDN. Cloud Run is origin.

Environments

EnvPurposeQuota / Cost ceiling
devLocal + per-engineer ephemeralMock YouTube + LLM. Zero external spend.
stagingIntegration + QAReal YouTube on a sandbox channel · LLM with $50/mo cap.
prodReal usersFull quotas · billing alerts at 50% / 80% / 100% of monthly budget.

Release

Trunk-based development. Every merge to main deploys to staging. Promotion to prod is a manual GitHub Action with required approvers — no auto-prod until we have a 30-day clean staging streak. Database migrations are forward-compatible by convention; no destructive change in the same release as code that depends on it (expand-then-contract).

Observability

The three signals + one

  • Logs — structured JSON, OpenTelemetry-shipped to Cloud Logging. Every log line carries tenant_id, trace_id, service.
  • Metrics — Prometheus-format, scraped by Cloud Monitoring. RED method (Rate / Errors / Duration) per service + USE for infra.
  • Traces — OpenTelemetry across services. A single comment's path from YouTube to alert should appear as one trace.
  • Business events — separate event stream feeding BigQuery: creator_signed_up, reply_approved, summary_rated. This is your product telemetry, not your engineering telemetry.

Scalability plan

A staged plan, not premature optimisation. Each tier has an explicit "exit signal" that triggers the next.

TierPostureExit signal
0–100 channelsSingle Postgres · Cloud Run · 1 classifier podp95 dashboard latency > 500ms or classifier queue lag > 5 min
100–1,000Read replica for dashboard · classifier HPA up to 5 pods · Pub/Sub batchingSingle Postgres write IOPS at 70% sustained
1,000–10,000Logical sharding by tenant_id · partitioned comments table by month · separate analytics replicaSingle-tenant viral-channel comment burst saturates classifier
10,000+Per-shard Postgres · regional active-active · dedicated tenant pools for top 1% creatorsBeyond the planning horizon of this doc
Anti-pattern

Do not Kubernetes-everything in tier 0. Cloud Run is enough for ten thousand creators. Ops complexity is a tax we don't pay until growth forces us to.

§ 14

Disaster recovery

  • RPO 15 minutes · RTO 2 hours for the MVP — published commitment.
  • Postgres: PITR enabled, automated daily snapshots, weekly snapshot copied to a second region.
  • Object store: cross-region replication for backups bucket only.
  • Pub/Sub: subscriptions retain undelivered messages for 7 days — safe replay window.
  • Quarterly game-day exercise: drop the prod replica, verify failover, restore a tenant from snapshot, time the whole thing. If you don't practise it, you don't have it.

Decisions that compound

  • One language per layer. Python for backend, TypeScript for frontend. Resist the urge to introduce Go for "performance" until measured.
  • Architecture Decision Records (ADRs). Every significant choice gets a numbered markdown file in /docs/adr/. Future-you and future-hires will thank you.
  • Feature flags from week one. A simple per-tenant flag table. Saves you twice in year one.
  • Type the entire bus. Pub/Sub event schemas as Pydantic models (Python) compiled to TypeScript. No untyped JSON crossing service boundaries.
  • Failure-mode tests. For each service, write one test that simulates: YouTube 403, DB connection drop, LLM timeout. The system should degrade gracefully, not crash.
§ 16

Path to YouTube partnership

A staged commercial plan. Architecture isn't only for engineers — it's evidence in the conversations you'll have with Google.

  1. Now → 3 months. Ship MVP. Don't contact YouTube. Apply to Google for Startups Cloud Program for credits.
  2. 3–6 months. 50 paying creators. Compile a public privacy policy, terms, and data-deletion runbook. Publish a public status page.
  3. 6 months. Submit for the YouTube API Services Compliance Audit. Budget 4–8 weeks. Use this architecture document as part of the submission.
  4. 8–10 months. Audit passed. Apply for quota extension citing audit verification. You are now meaningfully different from 99% of YouTube apps in the wild.
  5. 10–12 months. Reach out to YouTube Partner Program / Creator Relations with: paying users, audit pass, a data-driven case study (e.g. "creators using CreatorCore reduce comment-handling time by 64% — here's the methodology"). Now they listen.
  6. 12+ months. Inbound becomes possible. Speak at YouTube's creator summits. Partner integrations into YouTube Studio become a real conversation.
The reframe

You are not asking YouTube for a favour. You are showing them a tool that increases creator retention on their platform — a metric they care about more than almost anything. Build that case quantitatively before you book the meeting.
